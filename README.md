# Briefly summarize the DriverPass project. Who was the client? What type of system did they want you to design?
What did you do particularly well?
  
# The DriverPass project was a solution presented to us to help curb the rising number of DMV driving test failures by providing learners with the tools they need to pass. The client, Liam, owner of DriverPass,  wanted the DriverPass system to be an online platform that would allow users to access study materials, track their progress, and schedule driving lessons with in-car instructors. Further, Liam wanted the system to be updated from the DMV database so that his users would have access to the most recent study materials. On the administrative side of things, Liam requested that DriverPass maintain security over student data, allow certain driving instruction packages to be added or removed depending on driver availability, and allow him to track every appointment within the system.

# How did you interpret the user’s needs and implement them into your system design? Why is it so important to consider the user’s needs when designing?

  #To implement the user in my system design, I thought about what I would want from a system if I were trying to access it. Trustworthy security, robust technical support, and a smooth user interface are all things I look for when browsing a new-to-me system. Designing your system around the client's wants and needs is definitely essential, but including the user in that process is what makes well-made systems. There is no point in a system that no user wants to use; therefore, it is all the more important to consider the user's needs when designing a system. Some examples in my design that were designed around the user include the 3-tries-per-login sequence and the instructor-led driving training scheduling. The 3 attempts before the account is locked and requires a reset is how many systems handle user interactions to help prevent brute-force hacking attempts to gain access to the user's account. I implemented this in my design because any system that requires users to provide credentials to access it should have robust security measures in place to protect valuable user data. I also implemented a system that streamlines how users schedule, modify, or cancel their driving lessons with instructors. The reason is that ease of use and control over their system promote a healthy user-system relationship and keep them engaged. Without this in place, the system risks users encountering bugs or becoming frustrated as they try to find a workaround. 


# How do you approach designing software? What techniques or strategies would you use in the future to analyze and design a system?

  #Designing software is a rickety bridge for me. I think the best way to approach software design is to start slow and small and work my way up to the tree. The most important part of designing anything is to test profusely. I would use that as my basis: start small, don't overextend, and test everything. There are a couple of software design techniques, but for my specific approach, I recommend Top-Down Design. Starting with an idea of how the system should operate and then decomposing it into individual branches is the best way forward. This will result in the fewest issues in the end and produce a system that runs well and considers every possibility.
